# Trino 기반 Ignite / PostgreSQL / Redis 성능 평가 보고서

> 작성일: 2026-07-11
> 대상: `dev-cluster` (`ignite`, `cnpg`, `redis`, `trino` 네임스페이스)
> 작성 목적: Trino로 세 엔진을 묶어 운영할 때 실제로 뭐가 되고 안 되는지, 어느 조합이 이 프로젝트의
> 요구사항(OMOP 임상데이터 + 향후 대용량 스트리밍 데이터)에 맞는지 실측을 근거로 정리한다.

---

## 1. 배경 및 목적

이 프로젝트는 원래 Kafka로 들어오는 데이터를 Ignite에 hot buffer로 잠깐 올려두고, Spark가
주기적으로 Iceberg/S3로 내려보내며, Trino가 그 위에서 PostgreSQL·Redis까지 포함해 하나의 SQL
인터페이스로 federate하는 구조를 목표로 하고 있었다. 문제는 이 구성이 실제로 배포는 됐지만 한
번도 제대로 검증된 적이 없었다는 점이다 — 세 엔진에 같은 데이터를 넣고 같은 쿼리를 돌려본 적이
없었고, 재기동했을 때 데이터가 남는지, 장애가 나면 뭘 해야 하는지도 문서화돼 있지 않았다.

그래서 이번 작업은 크게 세 단계로 진행했다.

1. 실제로 동일한 데이터를 세 엔진에 넣고 정합성·성능을 비교한다.
2. 재기동/장애 상황에서 각 엔진이 어떻게 동작하는지 확인하고, 필요한 설정(persistence, 백업)을
   채워 넣는다.
3. 향후 예상되는 데이터 규모(하루 수백만 건)를 가정해 실제로 그 규모까지 넣어보고 병목을 찾는다.

---

## 2. 테스트 환경

| 항목 | 내용 |
|---|---|
| 클러스터 | `dev-cluster` (kubeadm, control-plane 1 + worker 3, 각 8vCPU/32GB) |
| 엔진 버전 | Trino 481, Apache Ignite 2.18.0, PostgreSQL 18(CloudNativePG 1.29.1 관리), Redis 7 |
| 초기 리소스 배분 | Ignite: cpu limit 4 / memory 30Gi, PostgreSQL: cpu limit 1 / memory 1Gi (기본값 그대로 방치돼 있었음) |
| 스토리지 | OpenStack Cinder CSI (`csi-cinder-sc-retain`) |

시작할 때는 세 엔진 다 기본 설정에 가까운 상태였고, 리소스 배분도 딱히 근거 없이 정해져 있었다.
이 부분이 나중에 결과 해석에 중요한 변수가 된다(6장 참고).

## 3. 테스트 범위

### 3-1. 3엔진 동일 데이터 적재 (OMOP 샘플, 112,184행)

병원 임상데이터 표준 모델(OMOP CDM) 샘플 CSV 10개를 Ignite·PostgreSQL·Redis에 동일한 스키마로
적재하고, Trino로 같은 쿼리를 돌려 행 수와 성능을 비교했다.

### 3-2. 재기동/휘발성 테스트

세 엔진의 Pod를 강제로 재기동시켜 데이터가 남는지 확인했다. "인메모리 DB"라는 이름만 보고
내구성을 판단할 수 없다는 걸 확인하는 게 목적이었다.

### 3-3. 900만 행 규모 부하 테스트

실제 예상 물량(하루 약 90만 건)의 10배를 가정해, 임의 스키마로 900만 행을 생성하고 Ignite·
PostgreSQL에 넣어 적재 속도·스토리지·쿼리 성능을 측정했다. Redis는 이 단계부터 의도적으로
제외했다(4장 참고).

### 3-4. 리소스 균등화 재검증

3-3에서 두 엔진의 리소스 배분이 애초에 달랐다는 문제를 인지하고, 동일 스펙으로 맞춘 뒤
PostgreSQL을 그 리소스에 맞게 튜닝해서 다시 비교했다.

### 3-5. 백업/복구 검증

PostgreSQL 백업(오브젝트 스토리지 연동)을 실제로 구성하고 온디맨드 백업이 성공하는 것까지
확인했다.

## 4. 주요 결과

### 4-1. Redis는 결과 발표 전에 이미 탈락시켰다

세 엔진 다 Trino로 접근하지만, Redis는 서버 자체가 조건절을 평가하는 기능이 없다. Trino가
`WHERE` 절이 붙은 쿼리를 보내면, Redis 커넥터는 결국 테이블에 속한 키를 전부 SCAN하고 값을
다 끌어온 다음 Trino 쪽에서 걸러낸다. 그 결과 112K행 규모 테스트에서 Redis는 **PK 조회조차
포함해 모든 쿼리 유형에서 가장 느렸다** (PK 조회 2.98초 vs Ignite 1.45초·PostgreSQL 1.51초).
데이터가 늘어날수록 이 격차가 좁혀질 이유가 없는 구조라서, 900만 행 테스트부터는 아예 대상에서
뺐다. (Redis 자체가 느리다는 뜻은 아니다 — Trino를 거치는 순간 Redis의 강점인 O(1) 키 조회를
못 살리는 구조라는 것.)

### 4-2. 재기동 테스트 — Ignite만 데이터가 사라졌다

`death` 테이블(26행)로 세 엔진을 강제 재기동해보니 PostgreSQL과 Redis는 데이터가 그대로 남았고,
Ignite만 테이블 자체가 사라졌다. 원인은 버그가 아니라 이 배포가 원래 "휘발성 hot buffer" 용도로
persistence를 꺼둔 상태였기 때문이다. Native persistence를 켜고 나서 재검증하니 재기동 후에도
데이터가 남는 것을 확인했다.

### 4-3. 900만 행 테스트 — 1차 결과 (리소스 불균형 상태)

임의로 만든 IoT 이벤트성 테이블(10컬럼, 900만 행)을 Ignite/PostgreSQL 각각의 네이티브 벌크로드
기능(Trino의 `INSERT`는 이 규모에서 너무 느려서 배제)으로 적재했다. 둘 다 목표 처리량(하루 900만
건 = 초당 약 104건) 대비 압도적으로 여유로웠다 (PostgreSQL 초당 약 30만 행, Ignite 초당 약 8만
4천 행).

성능 쪽은 예상과 다른 결과가 나왔다. `COUNT(*)`와 PK 조회는 Ignite가 확실히 빨랐는데, 카디널리티
5만짜리 `GROUP BY`는 오히려 **PostgreSQL이 2.6배 더 빨랐다.** 원인을 추적해보니 Trino로 만든
Ignite 테이블이 병렬도(`query_parallelism`) 기본값 1로 생성되어 있었다 — 파드에 CPU 4개가
할당돼 있는데도 단일 스레드로 900만 행을 순차 처리하고 있었던 것. 병렬도를 4로 맞춰서 재적재하니
GROUP BY가 12.6초에서 5.9초로 2.1배 개선됐다.

### 4-4. 900만 행 테스트 — 2차 결과 (리소스 균등화 후)

여기서 한 가지 문제를 더 짚었다 —애초에 두 엔진의 파드 리소스 자체가 달랐다(Ignite는 메모리
30Gi, PostgreSQL은 1Gi). 이 상태에서 "Ignite가 빠르다"는 결론은 사실 "Ignite에 리소스를 30배
더 줬다"는 것과 다름없었다. 두 파드를 동일 스펙(cpu 4 / memory 16Gi)으로 맞추고, PostgreSQL도
그 메모리를 실제로 쓰도록 튜닝(`shared_buffers` 128MB→4GB, `work_mem` 4MB→256MB, 병렬 워커
2→4)한 뒤 전체 쿼리셋(단순 카운트, PK 조회, 비-PK 필터, GROUP BY, JOIN)을 다시 돌렸다.

결과가 뒤집혔다. PostgreSQL은 튜닝 전 대비 거의 모든 쿼리에서 약 4배씩 빨라졌고, **이제 거의
모든 쿼리에서 PostgreSQL이 Ignite를 앞선다** — JOIN은 12배(1.46초 vs 17.27초), 단순 필터는
9배(0.50초 vs 4.58초) 차이가 났다. 반대로 Ignite는 메모리를 줄인 영향(힙 6g→4g, off-heap
23GB→10GB)으로 몇몇 쿼리가 오히려 더 느려졌다.

이 과정에서 부수적으로 두 가지를 확인했다. 첫째, Trino는 `COUNT`/`GROUP BY`와 달리 **JOIN은
소스 엔진으로 내려보내지 않고 Trino 자신이 처리한다** — 그래서 JOIN에서 유독 "Trino 경유"와
"엔진 직접 질의"의 차이가 크게 벌어졌다. 둘째, Ignite는 **병렬도가 다른 테이블끼리는 네이티브
JOIN 자체가 안 된다**(`Using indexes with different parallelism levels in same query is
forbidden`) — 운영 시 반드시 챙겨야 할 제약사항이다.

### 4-5. 백업 검증

PostgreSQL은 클러스터에 이미 있던 MinIO를 재사용해 백업 저장소를 구성했다. WAL 아카이빙이
정상 동작하는 것, 그리고 온디맨드 백업을 실행해 실제로 오브젝트 스토리지에 파일이 쌓이는 것까지
확인했다(약 53MB, 그 시점 데이터 전체 분량).

### 결과 요약표 (900만 행, 리소스 균등화 + 튜닝 후, 엔진 직접 질의 기준)

| 쿼리 | Ignite | PostgreSQL | 배수 |
|---|---|---|---|
| `COUNT(*)` | 0.94초 | 0.48초 | PostgreSQL 2배 |
| PK 조회 | 0.87초 | 0.53초 | PostgreSQL 1.6배 |
| 단순 WHERE 필터 (비-PK) | 4.58초 | 0.50초 | PostgreSQL 9배 |
| GROUP BY (카디널리티 5만) | 5.91초 | 1.07초 | PostgreSQL 5.5배 |
| JOIN | 17.27초 | 1.46초 | PostgreSQL 12배 |

## 5. 테스트 과정에서 발견한 이슈

작업 도중 실제로 부딪힌 문제들은 아래와 같다 (자세한 원인과 해결책은 `docs/lessons-learned.md`
1장 참고).

- Trino coordinator가 대량 INSERT 중 메모리 부족으로 죽음 → 메모리 상향, 배치 크기 축소
- Redis 키 포맷이 문서와 실제 동작이 달라서 신규 테이블이 항상 0행으로 조회됨
- Redis HSET에 공백 포함 값을 그냥 넘겨서 인자 파싱이 깨짐
- Ignite가 예전에 적용했던 설정 변경 때문에 재기동 시점에야 크래시(StatefulSet/ConfigMap 드리프트)
- Ignite의 두 가지 설정 버그(Kubernetes API 기반 노드 탐색 모듈 미로드, `tryStop` 프로퍼티 오류)
- nginx-ingress가 붙이는 헤더를 Trino가 기본적으로 거부해서 모든 쿼리가 406 에러
- CNPG postgres 컨테이너의 `/tmp`가 읽기전용이라 벌크로드 스테이징 경로를 바꿔야 했음
- Ignite CLI 도구(`sqlline`)가 비대화형 환경에서 자격증명 입력을 기다리며 멈춤

이 중 다수가 "데이터 양이 적을 때는 안 보이다가, 규모가 커지고서야 드러난 문제"였다는 공통점이
있다.

## 6. 결론 및 시사점

### 6-1. 이번 테스트만 놓고 보면 PostgreSQL이 이겼다

리소스를 공정하게 맞추고 기본 튜닝을 적용한 상태에서, 단일 노드·Trino 페더레이션이라는 조건
하에서는 PostgreSQL이 Ignite보다 거의 모든 쿼리에서 빨랐다. 여기서 얻을 수 있는 가장 중요한
교훈은 개별 수치보다 **"엔진 비교는 반드시 동일한 리소스 조건에서 해야 한다"**는 것이다 — 리소스
배분이 다르면 "어느 엔진이 빠른가"가 아니라 "어느 쪽에 리소스를 더 줬는가"를 측정하게 된다.

### 6-2. 그렇다고 Ignite를 걷어내는 게 정답은 아니다

이번 테스트는 단일 노드, 배치성 분석 쿼리라는 좁은 조건이었다. Ignite가 실제로 강점을 보이는
영역(여러 노드로 수평 확장했을 때의 처리량, 조인 키를 데이터와 함께 배치해서 셔플을 없애는
collocated join, Kafka 같은 스트림을 초당 수만 건 단위로 흡수하면서 동시에 SQL로 조회 가능한
상태 유지)은 이번 테스트 범위 밖에 있다. 오늘 결과로 "Ignite는 필요 없다"고 결론 내리는 건
성급하다 — 다만 지금처럼 단일 노드로 운영하면서 배치 분석 쿼리 위주로 쓸 거라면, 그 용도에는
PostgreSQL이 더 적합하다는 근거는 확보됐다.

### 6-3. Redis는 Trino 분석 경로에서 빼는 게 맞다

Redis는 여전히 빠른 키-값 저장소지만, Trino를 거쳐 SQL로 접근하는 순간 그 장점을 살릴 수 없는
구조적 문제가 있다. 애플리케이션이 직접 Redis 클라이언트로 붙는 캐시 용도로는 그대로 두고,
Trino 기반 분석 쿼리 대상에서는 계속 제외하는 걸 권장한다.

### 6-4. 리소스/설정 없이는 아무 결론도 못 낸다

이번 세션에서 가장 크게 배운 건 특정 엔진의 우열이 아니라, **설정과 리소스가 실제 데이터 규모에
맞춰져 있지 않으면 어떤 비교도 의미가 없다**는 점이다. Ignite도, PostgreSQL도 각각 한 번씩
"설정이 데이터 규모를 못 따라가서" 실제 성능보다 나쁘게 측정된 구간이 있었다.

### 6-5. 참고 — "실시간 집계"가 아니라 "실시간 조회"가 목적이라면

지금까지의 테스트는 대부분 `COUNT`/`GROUP BY` 같은 집계 쿼리 위주였다. 하지만 실제로 검토 중인
용도 중 하나는 성격이 다르다 — **컬럼 조건으로 파일이 어디 있는지 찾아서(경로 조회) 다운로드까지
연결하는 것**, 즉 `SELECT` + `WHERE` 필터가 대부분이고 `JOIN`이 가끔 섞이는 워크로드다. 이건
OLAP성 집계가 아니라 전형적인 OLTP성 point/range lookup이라, 지금까지의 "Ignite가 GROUP BY/JOIN
에서 밀렸다"는 결론을 그대로 가져다 쓸 수 없다 — 다만 그 밑에 깔린 교훈(엔진 선택보다 인덱스·
리소스 튜닝이 결과를 더 좌우한다)은 똑같이 적용된다.

이런 형태(파일 메타데이터 카탈로그)에 대한 현재 업계 트렌드는 대략 이렇게 정리된다.

- **파일 실체와 메타데이터를 분리한다.** 실제 파일은 S3/MinIO 같은 오브젝트 스토리지에 두고,
  DB에는 경로·크기·체크섬·태그·타임스탬프 같은 메타데이터만 저장한다. 큰 파일을 DB에 BLOB으로
  직접 넣는 방식은 점점 안 쓰는 추세다. (이 프로젝트에서 이미 MinIO를 PostgreSQL 백업 대상으로
  써봤으니, 같은 오브젝트 스토리지를 파일 저장소로 재사용하는 것도 자연스러운 확장이다.)
- **다운로드는 DB/앱을 거치지 않는다.** 메타데이터 조회로 경로를 찾아낸 다음, 그 경로에 대한
  presigned URL(S3) 또는 CDN URL을 발급해서 클라이언트가 오브젝트 스토리지에서 직접 받게
  하는 구조가 표준이다 — DB는 "어디 있는지"만 답하고, 실제 파일 전송 트래픽은 완전히 분리된다.
- **구조화된 필터+조인 쿼리(이번 케이스와 정확히 일치)는 여전히 인덱스 건 RDBMS가 제일 유리하다.**
  B-tree 인덱스만 걸려 있으면 point/range lookup은 데이터가 아무리 늘어나도 거의 일정한 응답
  시간을 유지하고, JOIN도 RDBMS의 핵심 강점 영역이다. 이번 보고서의 4-4/6-1에서 확인했듯
  PostgreSQL을 제대로 튜닝하면 이 워크로드에서 이미 우세했는데, **정작 이번 테스트에서는
  인덱스를 한 번도 걸어보지 않았다** — 이 케이스라면 인덱스 적용이 가장 먼저 시도해볼 지점이다.
- **자유 텍스트/태그 검색처럼 필터 조합이 다양하고 유연해야 하면** Elasticsearch/OpenSearch,
  Meilisearch/Typesense 같은 검색엔진을 보조 인덱스로 두고 원본 RDBMS에서 CDC로 동기화하는
  구성이 늘어나는 추세다. 다만 질문에서 설명한 "컬럼 조건 필터 위주"라면 지금 단계에서는 RDBMS
  인덱스만으로도 충분할 가능성이 높다.
- **Redis 같은 KV 캐시는 "파일 ID를 이미 알고 있고 경로만 빨리 받고 싶다"는 좁은 케이스에서는
  여전히 유용하다.** 하지만 이번 질문처럼 `WHERE` 필터가 들어가는 순간, Redis 단독으로는
  대체가 안 된다 — `lessons-learned.md` 2-1에서 확인한 "Redis는 서버사이드 필터링이 없다"는
  구조적 한계가 Trino를 거치든 애플리케이션이 직접 붙든 똑같이 적용되기 때문이다.

**결론**: 이 유스케이스만 놓고 보면, 지금까지 테스트한 세 엔진 중 **인덱스를 건 PostgreSQL이
가장 잘 들어맞는 조합이다** — 오히려 집계 위주 테스트보다 더 명확하게 맞아떨어진다. Redis는
파일 ID 기반 캐시로 보조적으로 붙일 수는 있지만 주 조회 경로는 아니고, Ignite의 강점(분산 집계,
스트림 흡수)은 이 워크로드에서 딱히 필요하지 않다.

## 7. 향후 계획

- Ignite를 2~3개 노드로 늘려서 수평 확장 시 처리량/쿼리 성능이 실제로 개선되는지 검증
- 조인 키를 affinity key로 맞춘 collocated join으로 Ignite JOIN 성능 재검증
- PostgreSQL/Ignite 둘 다 비-PK 필터·조인 컬럼에 인덱스를 걸고 재측정
- Trino worker를 실제로 추가해서 JOIN 쿼리 성능이 개선되는지 확인 (지금까지는 coordinator
  단독 구성으로만 테스트함)
- 이번에 줄인 Ignite 메모리(16Gi) 배분을 데이터 규모에 맞게 재조정
- 상시 연결 클라이언트 기반으로 재벤치마크 (현재 측정은 매 쿼리마다 CLI를 새로 띄우는 방식이라
  고정 오버헤드가 섞여 있음)

세부 수치와 재현 절차는 `docs/test-results-summary.md`, `docs/lessons-learned.md`,
`docs/airgap-full-test-runbook.md`에 정리되어 있다.
